# 咩咩羊采集插件优化增强审阅与实施建议

## 1. 文档信息

- 文档日期：2026-07-10
- 审阅范围：`shopify-product-collector/`、`tools/`、Manifest、现有任务与测试文档
- 当前版本：Manifest `1.0.12`
- 文档目标：把本轮代码审阅结果整理成可执行、可验收、可持续维护的优化路线

## 2. 总体结论

插件目前已经具备较完整的商品采集与 Shopify CSV 生产能力，包括：

- Shopify、Amazon、WooCommerce、Shopline、Shoplazza 和通用 DOM 兜底采集。
- 商品页、列表页、Collection 页发现与跨页扫描。
- 单品草稿、批量队列、图片检查、变体编辑与导出前校验。
- 内容脚本按需注入，减少普通网页的常驻性能负担。
- 图片检查并发限制与缓存、图片和变体分页、表单增量保存。
- Shopify CSV 单品与批量导出。

当前阶段不缺功能入口，下一阶段最有价值的工作是强化以下三项基础能力：

1. 数据正确性：避免生成不存在的变体、错误币种价格或错误域名规则。
2. 任务可靠性：确保批量停止、超时、恢复和持久化行为可预测。
3. 工程可维护性：建立可复现测试、模块边界、数据版本和升级机制。

## 3. 已确认的良好设计

以下能力建议保留并继续作为后续改造的基础：

- `content.js` 使用初始化标记避免重复注入。
- `popup.js` 通过 `sendMessageWithInjection` 按需注入内容脚本。
- 商品采集按 JSON-LD、Shopify、Meta、站点规则、平台规则和通用 DOM 分阶段兜底。
- 批量采集优先尝试 Shopify Ajax 商品接口，失败后再创建后台标签页。
- 图片检查限制并发数，并复用 URL 级缓存和 in-flight Promise。
- 图片、变体、批量商品均具备风险提示与导出前校验。
- 批量队列中的 `running` 状态在恢复时会被归一化为 `pending`，具备基础断点恢复意识。
- CSS 已开始按 tokens、base、components、popup、utilities 分层，并统一设计 token。

## 4. P0：数据正确性与可靠性

### 4.1 使用真实变体矩阵，避免生成虚假组合

#### 现状

- `content.js:3828` 的 `buildDomVariantsFromGroups` 对所有 DOM 选项做笛卡尔积。
- `content.js:5532` 的 `buildAmazonVariants` 同样组合所有 Amazon 选项组。
- 当前逻辑无法判断某个颜色和尺寸组合是否真实存在、是否可售或是否有独立 SKU。
- 选项数量较多时，组合数会快速膨胀，增加内存、渲染、存储和 CSV 行数。

#### 风险

- 导出并不存在的商品变体。
- 把一个选项的价格、图片或 ASIN错误地复制到其他组合。
- 大型选项组导致采集长时间卡顿。

#### 建议方案

1. Shopify 优先使用 Ajax JSON 中的真实 `variants` 列表。
2. Amazon 优先解析 ASIN 与选项关系数据，不将不同组选项直接自由组合。
3. 通用站点通过以下方式获取真实组合：
   - 读取 `variantId`、SKU、JSON 状态或表单隐藏字段。
   - 必要时逐项切换选项，等待页面状态变化后记录真实选中组合。
   - 通过 variant ID、SKU 或完整选项键去重。
4. 通用兜底增加最大组合数，例如 100；达到上限时停止并展示风险。
5. 为无法验证真实性的结果增加 `variantConfidence: inferred`，导出前明确提示。

#### 验收标准

- 不生成页面中无法选择的变体组合。
- 变体数超过上限时插件不中断、不假死，并展示可理解的提示。
- Shopify 真实变体数量与 `/products/{handle}.js` 返回数量一致。
- Amazon 变体至少能按 ASIN 去重，不产生重复或交叉虚假组合。

### 4.2 建立可靠的持久化层

#### 现状

- `popup.js:722` 的 `chromeStorageGet`、`chromeStorageSet`、`chromeStorageRemove` 没有检查 `chrome.runtime.lastError`。
- `content.js` 中站点规则和图片区域存储也存在同类情况。
- `popup.js:955` 将完整批量队列、商品、图片、变体和日志写入单个 `spc:batch-queue:v1` 键。
- 草稿持续累积，没有容量提示、过期时间或 LRU 清理。
- 多个并发 worker 会重复写入整个批量对象，存在旧快照晚到后覆盖新快照的可能。

Chrome `storage.local` 默认容量为 10 MB，写入超限时回调模式通过 `runtime.lastError` 报告失败。

#### 建议方案

1. 统一实现 `StorageRepository`：
   - 所有 Chrome Storage 调用正确 resolve/reject。
   - 写入失败时在 UI 显示具体错误，不再提示“已保存”。
   - 为存储对象增加 `schemaVersion`、`revision`、`createdAt` 和 `updatedAt`。
2. 使用串行写队列或 latest-write-wins 调度器，避免并发覆盖。
3. 设置和小型索引保留在 `chrome.storage.local`。
4. 商品草稿、批量商品结果和历史记录迁移到 IndexedDB，按商品独立记录。
5. 增加容量管理：
   - 使用 `getBytesInUse()` 展示占用。
   - 草稿设置保留期限或最大数量。
   - 支持清理已完成批次、失败日志和图片检查缓存。
6. 增加 v1 到后续版本的数据迁移测试。

#### 验收标准

- 模拟 `QUOTA_BYTES` 错误时，用户可以看到保存失败，内存状态不被误标为已持久化。
- 并发完成多个商品后，重新打开侧边栏不会丢失最新结果。
- 旧版 `spc:batch-queue:v1` 数据可以迁移或安全恢复。
- 大批量队列不会因为单个超大存储项导致整体无法保存。

### 4.3 分离活动标签页与正在编辑的草稿上下文

#### 现状

- 侧边栏初始化后没有监听 `chrome.tabs.onActivated` 或当前标签页导航变化。
- `currentTabUrl` 同时承担活动页面地址和正在查看的批量商品地址。
- `popup.js:4004` 的站点规则读取依赖 `currentTabUrl`。
- 规则测试和取点操作又使用实时活动标签页，可能出现旧域名规则作用于新页面的问题。

#### 建议方案

1. 拆分状态：
   - `activeTabContext`：当前浏览器标签页的 id、URL、域名和标题。
   - `displayedDraftContext`：当前编辑的单品草稿或批量商品。
2. 监听：
   - `chrome.tabs.onActivated`
   - `chrome.tabs.onUpdated`
   - `chrome.windows.onFocusChanged`（必要时）
3. 实现统一的 `switchActiveTabContext()`：
   - 保存当前未提交表单。
   - 更新页面信息和域名徽标。
   - 加载新域名站点规则。
   - 恢复对应商品草稿。
4. 站点规则测试、保存、清除和取点前再次校验 tab id 与域名。
5. 批量商品预览模式明确标记为“正在查看队列商品”，避免与活动页面混淆。

#### 验收标准

- 切换标签页后，页面信息、规则域名和采集目标同步更新。
- 不会把 A 域名的选择器保存到 B 域名。
- 查看批量商品时切换标签页，不会意外覆盖批量草稿。

### 4.4 重构批量任务生命周期

#### 现状

- `popup.js:2891` 使用 HEAD 请求预检查链接。
- 部分站点会对 HEAD 或扩展请求返回 403，但正常浏览器页面仍然可访问。
- `collectBatchItem` 当前把所有预检查 HTTP 错误视为阻断错误。
- 停止操作只停止领取新任务，正在运行的页面采集、图片检查和 Amazon 翻页会继续。
- `withTimeout` 只让外层 Promise 超时，没有取消底层任务。
- `waitForBatchTabLoad` 在创建标签页之后才监听 `tabs.onUpdated`，可能错过快速页面的 `complete` 事件。
- 页面加载等待超时和正常加载均以 resolve 结束，无法准确分类。

#### 建议方案

1. 预检查策略：
   - 404、410 可以作为强错误。
   - 401、403、429 和 5xx 作为警告，继续尝试真实标签页。
   - 已手动预检查且结果未过期时，正式采集不重复请求。
2. 为每个任务建立 `BatchJobContext`：
   - `AbortController`
   - `tabId`
   - `startedAt`
   - 当前阶段
3. 停止时取消所有 controller，并关闭插件创建的后台标签页。
4. `waitForBatchTabLoad` 首先调用 `chrome.tabs.get(tabId)` 检查当前状态，再注册监听。
5. 同时监听 `tabs.onRemoved`，区分加载完成、超时、被关闭和导航失败。
6. `startBatchCollection` 使用整体 `try/finally`，确保 UI 状态、按钮和临时标签页始终恢复。

#### 验收标准

- 点击停止后不再继续创建新标签页，活动任务能在合理时间内取消。
- HEAD 403 但页面可访问的商品仍会继续采集。
- 快速加载页面不会无故等待完整超时时间。
- 任务异常后批量按钮不会永久停留在“采集中”。

### 4.5 限制 Shopify 同类商品探测成本

#### 现状

- `content.js:3137` 的 `collectShopifySiblingProducts` 顺序处理所有候选链接。
- 每个候选可能依次尝试 `.js` 和 `.json`，单次请求超时为 5.5 秒。
- 候选来自属性和脚本文本，目前缺少数量上限和整个阶段的总时限。

#### 建议方案

- 只保留评分最高的 8 个候选。
- 设置总预算，例如 8 秒，而不是每个候选独立消耗完整超时。
- 使用 2 至 3 个并发 worker。
- 先按标题相似度、handle、规格词和主商品信息预筛选，再发请求。
- 将“跨商品合并为变体”作为可关闭的高级设置，并在结果中标明合并来源。

#### 验收标准

- 单品采集不会因几十个候选链接延迟数十秒。
- 超过总预算后保留已确认结果并正常返回。
- 不会把样品、赠品或无关同系列商品误合并为变体。

## 5. P1：测试与架构

### 5.1 统一测试工程

#### 当前验证结果

- `manifest.json` 可以正常解析。
- `background.js`、`content.js`、`csv.js`、`popup.js` 均通过 `node --check`。
- 仓库包含 8 个 `tools/run-*.mjs` 测试脚本。
- 当前仓库没有 `package.json`、依赖锁文件或统一测试命令。
- 在未配置外部 Playwright 的干净环境运行时，测试会因仓库未声明 `playwright` 而无法启动。
- 本轮通过系统临时目录安装 Playwright 1.61.1 后，8 个现有回归脚本全部通过。
- 不同脚本重复实现 Playwright 查找、Chrome mock、Storage mock 和断言工具。
- 大部分测试通过 mock API 或向普通页面注入脚本运行，没有覆盖真实未打包扩展生命周期。

#### 建议方案

1. 新增 `package.json` 和锁文件，固定 Playwright 版本。
2. 提供统一命令：
   - `npm run check`
   - `npm run test:unit`
   - `npm run test:e2e`
   - `npm test`
3. 抽取 `tools/test-utils.mjs`：
   - 浏览器启动与 fallback。
   - Chrome API mock。
   - Storage mock 和错误注入。
   - 测试页面 fixture 与通用断言。
4. 将价格解析、URL 规范化、CSV 行转换、变体组合、校验规则抽成纯函数测试。
5. 增加真实扩展冒烟测试：
   - 使用 persistent context 加载未打包扩展。
   - 打开测试商品页。
   - 点击扩展 Action 或打开 Side Panel。
   - 验证消息、按需注入、存储和下载完整链路。
6. 在 GitHub Actions 中运行语法检查和稳定 fixture 测试；真实网站测试单独运行，避免外部页面变化阻塞主 CI。

#### 验收标准

- 新克隆仓库执行一次依赖安装后即可运行全部稳定测试。
- CI 能发现 Manifest、消息协议、CSV 和核心采集回归。
- 测试不依赖开发者机器上的全局 Playwright 路径。

## 6. P1：产品增强

### 6.1 字段级来源与置信度

当前仅显示整个商品的总体来源，但实际标题、价格、图片、品牌和变体可能来自不同采集层。

建议为字段保留：

```javascript
{
  value: "24.99",
  source: "json-ld",
  confidence: 0.92,
  alternatives: [
    { value: "29.99", source: "dom-compare-price", confidence: 0.61 }
  ]
}
```

UI 可以默认显示最终值，并提供“来源详情”查看候选。低置信字段进入导出风险列表。

### 6.2 币种识别与防错

`content.js` 已能从 JSON-LD 获取 `currency`，但侧边栏没有展示，导出也没有验证源币种与目标 Shopify 店铺币种是否一致。

建议：

- 显示源币种、来源和识别置信度。
- 允许设置目标店铺币种。
- 币种不一致时默认阻止静默导出或给出强提醒。
- 汇率转换必须由用户明确启用，并保存汇率来源、时间和计算前价格。
- 改进欧洲小数与千位分隔符解析，例如正确区分 `1.234,56` 和 `1,234.56`。

### 6.3 站点规则健康度和迁移

建议为站点规则增加：

- 规则版本、创建时间、最后成功时间、最近命中率。
- 每个字段多个 fallback selector。
- 域名和路径级作用范围。
- 规则导入、导出和备份。
- 页面结构变化后的失效提示和重新取点入口。
- 规则测试结果快照，便于定位网站改版。

### 6.4 历史批次与商品草稿库

现有产品规划已经包含批量历史、重新导出、继续采集、商品草稿库和搜索。建议以 IndexedDB 重构为前置，而不是继续扩大单个 Chrome Storage 对象。

建议去重标识优先级：

1. Shopify handle 与站点域名。
2. Amazon ASIN。
3. SKU、GTIN、UPC 或 barcode。
4. 规范化 URL。
5. 标题与品牌的低置信相似匹配。

### 6.5 导出能力

- 保留当前 Shopify 新式字段模板，并增加官方样例契约测试。
- 统一使用 UTF-8，并按 Shopify 当前建议评估 LF 换行。
- 增加 CSV 公式注入风险检测；可提供“Shopify 原始导入”和“表格安全预览”两种模式。
- 增加导出前表格预览、列选择和字段映射。
- 后续扩展 JSON、Excel、WooCommerce、Shopline、Shoplazza 模板。
- 大批量导出支持按商品数或文件大小拆分。

## 7. P2：权限、安全与发布准备

### 7.1 收敛主机权限

Manifest 当前声明：

```json
"host_permissions": ["<all_urls>"]
```

该权限满足任意域名批量采集，但会增加安装警告和审核敏感度。

建议：

- 当前活动页继续使用 `activeTab + scripting`。
- 任意批量域名改为 `optional_host_permissions`，用户添加域名或开始批量任务时按需授权。
- 对受限页面、浏览器内部页面和无权限域名提供明确错误提示。
- 评估通过后台消息访问存储，并将 `storage.local` 访问级别限制为可信扩展上下文。

### 7.2 CSV 与页面数据安全

- 页面标题、SKU、标签和描述均属于不可信输入。
- DOM 输出继续使用 `textContent`，避免把采集内容直接写入界面 `innerHTML`。
- 对为解析 HTML 而使用的 detached element 保持不挂载策略。
- CSV 导出检测以 `=`, `+`, `-`, `@` 开头的潜在公式字段。
- Shopify 导入格式与 Excel 安全格式不能盲目共用转义规则，应分别提供模式。

### 7.3 发布信息

正式进入 Chrome Web Store 前建议补充：

- 最低 Chrome 版本说明。
- 权限用途说明。
- 隐私政策和“不上传到第三方服务器”的实际数据流说明。
- 更新日志和版本迁移说明。
- 失败反馈入口和诊断信息导出。

## 9. 建议新增的核心测试场景

1. Storage 超限、权限拒绝和回调 `runtime.lastError`。
